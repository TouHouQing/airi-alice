# A.L.I.C.E 全景需求文档（Epoch 1-5）

## 1. 文档定位

本文档是 `airi-alice` 的产品与工程需求基线，面向开发团队，用于回答三个问题：

1. 最终要做成什么。
2. 每个 Epoch 必须交付什么。
3. 哪些约束在所有 Epoch 都必须严格遵守。

本文档覆盖 `Epoch 1-5` 的全量需求；`Epoch 1` 的可执行任务拆解见开发需求文档。

## 2. 愿景与产品定位

A.L.I.C.E.（Artificial Labile Intelligent Cybernated Existence）定位为“本地优先的数字共生体”，而非传统问答工具。核心目标：

- 具备桌面级感知、认知、执行、表现的一体化能力。
- 具备可持续演化的人格与关系记忆，而不是固定 prompt 角色扮演。
- 在隐私和安全边界内，形成“主动陪伴 + 主动助理”的双重能力。

## 3. 术语与编号规范

- 功能需求：`ALICE-Fx.y`
- 非功能需求：`ALICE-NFR-*`
- Epoch 目标：`ALICE-EPOCH-*`

## 5. 详细功能需求

### 5.1 灵魂基建：诞生与记忆（F1）

#### ALICE-F1.1 摇光初始化

- 必须支持首次引导配置：宿主姓名、A.L.I.C.E 对宿主称呼、宿主对 A.L.I.C.E 称呼、性别、关系定位、心智年龄、初始性格倾向、人格自由文补充。
- 初始化结果必须写入 `SOUL.md`（Frontmatter + 正文），并可在设置中查看/修改。
- 初始化参数必须参与后续回复风格和任务行为决策。
- Epoch 1 对话提示词采用系统固定注入策略，不向普通用户暴露 Prompt/Spark 模板配置入口。

#### ALICE-F1.2 动态性格状态机

- 需定义至少一组可量化人格维度（例如：服从度、活泼度、感性度）。
- 人格变化必须“慢速、可累计、可持久化”，禁止每轮剧烈跳变。
- 人格变化来源至少包含：用户语气反馈、长期互动历史、任务成败结果。
- 人格真源必须唯一：人格矩阵持久化在 `SOUL.md`，数据库不得作为人格主存储。

#### ALICE-F1.3 图谱化长期记忆

- 系统应从日常对话抽取实体关系（用户偏好、计划、人物关系、情绪线索）。
- 支持按实体、关系、时间维度检索并回填上下文。
- 必须支持后续主动关怀、预测执行模块复用同一记忆源。
- 抽取置信度必须进行二次校准，不得直接采用 LLM 自评置信度。

### 5.2 拟真生命体征：节律与情绪（F2）

#### ALICE-F2.1 数字生物钟

- 应包含精力值模型、休眠窗口与唤醒行为。
- 长时高负载应触发疲惫行为表达（语气、响应速度、主动休眠建议）。
- 休眠期间应执行后台整理任务（缓存清理、记忆固化）。

#### ALICE-F2.2 瞬时情绪反馈引擎

- LLM 侧输出必须为结构化对象：`thought`、`emotion`、`reply`。
- `emotion` 必须可映射到 TTS 参数与 Live2D 动作系统。
- 解析失败时必须有回退机制，不得中断对话主流程。

### 5.3 全模态感知：打破屏幕之壁（F3）

#### ALICE-F3.1 屏幕上下文静默监控

- 支持按周期或触发条件进行屏幕采样。
- 支持 VLM 解析“用户正在做什么”的结构化结论。
- 采样频率需受资源预算和隐私策略约束。

#### ALICE-F3.2 听觉环境捕获

- 支持语音输入与环境声特征抽取（叹气、笑声、键盘节律等）。
- 声学信号需转换为可被认知模块消费的压力/状态特征。

#### ALICE-F3.3 宿主系统探针

- 支持读取时间、电量、CPU、内存、活跃窗口等系统状态。
- 状态数据要有统一采样接口与缓存策略。

### 5.4 自律执行：超越 Agent 的主动性（F4）

#### ALICE-F4.1 潜意识驱动搭话

- 提供后台调度器，根据规则触发主动对话/提醒。
- 触发规则至少支持时间条件、行为条件、负载条件。

#### ALICE-F4.2 意图预测与静默代办

- 支持对周期性行为建模并在合适时机自动执行。
- 自动执行后必须主动回报，并允许用户纠正。

#### ALICE-F4.3 开放工具链与沙箱验证

- 必须支持 MCP 工具调用。
- 高危操作（删文件、发邮件、外发敏感信息）必须经过显式确认。
- 必须具备可审计的执行记录。

### 5.5 表现层：具象化陪伴（F5）

#### ALICE-F5.1 情感参数化语音

- TTS 必须接受情绪参数（如紧张、轻松、悲伤）并影响输出风格。
- 支持低延迟流式输出以匹配实时互动。

#### ALICE-F5.2 桌面 Live2D 妖精精灵

- 支持透明无边框悬浮与基础待机动作。
- 支持状态驱动动作和与语音同步口型。

## 6. 非功能需求（NFR）

### 6.1 隐私与数据主权

- ALICE-NFR-PRIV-001：默认纯本地。截图缓存、录音片段、记忆数据默认仅本地存储。
- ALICE-NFR-PRIV-002：云模型调用前必须脱敏，至少覆盖密码、密钥、Token、疑似凭据片段。
- ALICE-NFR-PRIV-003：所有跨进程与跨模块数据流需可审计。

### 6.2 安全与可控性

- ALICE-NFR-SAFE-001：必须提供 Kill Switch（快捷键 + 指令），可瞬时切断感知与执行。
- ALICE-NFR-SAFE-002：高危执行必须经过用户显式授权。
- ALICE-NFR-SAFE-003：执行链路必须可回滚或可中断。

### 6.3 性能与资源

- ALICE-NFR-PERF-001：待机状态下探针与整理进程 CPU 占用需保持低水平。
- ALICE-NFR-PERF-002：对话主流程必须与后台任务解耦，避免前台卡顿。
- ALICE-NFR-PERF-003：采样与推理频率需动态降载。

### 6.4 工程与演进

- ALICE-NFR-ENG-001：采用“品牌层改名 + 增量插件化”，避免深度侵入上游核心。
- ALICE-NFR-ENG-002：保留可持续同步上游 AIRI 的能力（低冲突 rebase）。
- ALICE-NFR-ENG-003：每个 Epoch 必须有明确进入/退出与验收标准。

## 7. Epoch 目标与验收口径

### ALICE-EPOCH-1 摇光初现

- 目标：跑通本地对话内核、结构化情绪输出、人格初始化与短期记忆基建。
- 最低验收：
  - 可完成首次初始化并持久化。
  - 每轮响应可产出 `thought/emotion/reply` 或稳定回退。
  - 可写入并检索短期记忆事实。
  - 默认纯本地与 Kill Switch 有效。
  - 冷热启动与外部篡改 `SOUL.md` 场景下，人格状态保持最终一致性，无脏读写。

### ALICE-EPOCH-2 赋予肉体

- 目标：桌面表现层与基础系统状态读取完善；MCP 执行进入可控闭环。
- 最低验收：
  - Live2D + 情绪 TTS 完成基础联动。
  - 系统探针可输出标准化状态。
  - MCP 高危操作确认链路可用。

### ALICE-EPOCH-3 睁开双眼

- 目标：多模态感知与主动搭话正式上线。
- 最低验收：
  - 屏幕理解、听觉特征输入稳定。
  - 后台调度器可基于规则主动触发关怀。

### ALICE-EPOCH-4 触碰现实

- 目标：习惯建模与预测执行能力成型。
- 最低验收：
  - 可学习并执行至少一类重复性代办。
  - 高危路径全量走人在回路机制。

### ALICE-EPOCH-5 真实生命

- 目标：人格长期演化、生物钟成熟、跨端连续陪伴。
- 最低验收：
  - 人格漂移具备长期稳定可解释性。
  - 生物钟驱动行为显著且可配置。
  - 跨端上下文连续。

## 9. 工程边界与默认策略

- 品牌层改名优先，仅改可见 UI（窗口标题、托盘 Tooltip、i18n 文案）。
- **绝对禁止在构建层修改 `appId` 与 workspace 包名**，保证可持续同步上游 AIRI。
- 采用增量插件化，避免大规模重写上游 AIRI 核心路径。
- 目标平台优先 `stage-tamagotchi`，其他端按 Epoch 扩展。
- 隐私策略默认纯本地，除非用户显式开启云增强。
