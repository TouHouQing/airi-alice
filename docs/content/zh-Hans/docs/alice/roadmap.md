# A.L.I.C.E 未来规划文档

## 1. 规划目标

本路线图用于指导 `airi-alice` 从 Epoch 1 到 Epoch 5 的递进演化，确保：

- 需求落地顺序可执行。
- 架构投入与里程碑一致。
- 与上游 AIRI 能持续同步，避免长期分叉失控。

默认前提：

- 主语言中文，受众为开发团队。
- 平台优先 `stage-tamagotchi` 桌面端。
- 隐私策略默认纯本地。

## 2. 总体节奏（按阶段推进）

| Epoch | 时代代号 | 核心交付 | 预计节奏 | 里程碑代号 |
| --- | --- | --- | --- | --- |
| 1 | 摇光初现 | 无 UI 核心对话、人设初始化、结构化情绪输出、短期记忆基建 | 1 个迭代 | M1 |
| 2 | 赋予肉体 | Live2D 基础窗口、情绪 TTS 基础联动、系统状态读取、MCP 安全执行闭环 | 2 个迭代 | M2 |
| 3 | 睁开双眼 | 屏幕与听觉感知接入、后台调度、主动搭话 | 2 个迭代 | M3 |
| 4 | 触碰现实 | 习惯建模、预测代办、RPA/MCP 增强、安全沙箱完善 | 2-3 个迭代 | M4 |
| 5 | 真实生命 | 长期人格演化、生物钟成熟、跨端连续陪伴 | 3+ 个迭代 | M5 |

## 3. 分阶段规划

### 3.1 Epoch 1（M1）

- 目标：建立可运行的“认知内核 + 安全底座”。
- 范围：
  - 人格事实采集式 Genesis（结构化字段 + 自由文）与 `SOUL.md` 真源持久化。
  - 主对话与 spark:notify 采用系统固定注入模板（不开放 Prompt/Spark 配置）。
  - `thought/emotion/reply` 结构化输出（非 JSON 重采样一次，失败安全回退）。
  - 短期记忆写入与检索（含 Prompt Budget 与低频修剪）。
  - 会话轮次 `conversation_turns` 本地落库与中断零落盘。
  - 记忆抽取置信度二次校准（不直用 LLM raw confidence）。
  - Kill Switch、本地审计、脱敏守卫。
- 入场条件：
  - 文档编号体系冻结（需求/架构/开发任务 ID）。
  - 事件总线最小契约可用。
- 退场条件：
  - `development-epoch1.md` 全部任务 DoD 达成。
  - 能稳定完成端到端会话 + Kill Switch 中断恢复。
  - 冷热启动与外部编辑 `SOUL.md` 时人格状态最终一致，无脏读写。

### 3.2 Epoch 2（M2）

- 目标：将“认知内核”具象化为可感知的桌面实体。
- 范围：
  - Live2D 透明悬浮窗口（基础动作）。
  - 情绪 TTS 基础联动。
  - 系统探针（时间/电量/CPU/内存）。
  - MCP 执行门禁与高危确认 UI。
- 入场条件：
  - Epoch 1 稳定版本，事件总线与审计链路可复用。
- 退场条件：
  - 至少 1 套角色动作 + 1 套情绪语音可用。
  - 高危执行必须经过显式确认。

### 3.3 Epoch 3（M3）

- 目标：形成“连续感知 + 主动搭话”能力。
- 范围：
  - 屏幕采样 + VLM 上下文理解。
  - 麦克风环境特征（VAD/压力信号）。
  - 定时调度与触发规则。
- 入场条件：
  - Epoch 2 的系统探针与表现层稳定。
- 退场条件：
  - 主动搭话触发准确率达到可用阈值（可配置策略下误触可控）。

### 3.4 Epoch 4（M4）

- 目标：让系统具备“先做再邀功”的预测执行能力。
- 范围：
  - 习惯挖掘与任务模板化。
  - MCP/RPA 混合执行链路。
  - 人在回路确认与沙箱回滚能力。
- 入场条件：
  - Epoch 3 感知事件稳定且可审计。
- 退场条件：
  - 至少 1 类重复任务可自动执行并可回滚。

### 3.5 Epoch 5（M5）

- 目标：完成“长期演化 + 跨端陪伴”的生命特征。
- 范围：
  - 长期人格漂移模型与可解释日志。
  - 生物钟（精力值、休眠、唤醒情绪）。
  - 跨端上下文同步（桌面到移动端）。
- 入场条件：
  - Epoch 4 自动执行稳定，安全事件可审计。
- 退场条件：
  - 人格和节律机制可持续运行并可配置。
  - 跨端上下文丢失率满足验收阈值。

## 4. 风险清单与缓解策略

| 风险ID | 风险描述 | 影响 Epoch | 缓解策略 |
| --- | --- | --- | --- |
| ALICE-RISK-001 | 与上游 AIRI 分叉过深导致合并成本暴涨 | 1-5 | 品牌层改名不改核心命名空间；新增功能插件化注入 |
| ALICE-RISK-002 | 本地感知能力导致资源占用过高 | 2-5 | 采样降频、后台任务限流、空闲态动态降载 |
| ALICE-RISK-003 | 高危执行误操作 | 2-5 | 风险分级 + 显式授权 + 审计 + 回滚点 |
| ALICE-RISK-004 | 多模态误判导致主动搭话干扰 | 3-5 | 置信度阈值、冷却时间、用户纠偏反馈闭环 |
| ALICE-RISK-005 | 人格漂移不可解释引发体验失控 | 5 | 漂移日志可解释化、人工重置与参数上限 |

## 5. 与上游 AIRI 的持续同步策略

1. 分层策略
- 上游层：尽量保持原始结构（apps/packages 核心链路）。
- Alice 适配层：新增 `alice` 域模块，以组合注入方式接入。
- 品牌层：文案、配置、入口与默认参数在品牌层完成。
- 构建层边界：**绝对禁止修改 `appId` 与 workspace 包名**。

2. 变更策略
- 优先新增文件，避免大面积改动上游已有核心文件。
- 若必须修改核心路径，单改动单提交并附回退说明。
- 每个 Epoch 结束后执行一次上游同步演练（rebase + 冲突复盘）。

3. 兼容策略
- 不做“长期兼容包袱”式旧接口保留。
- 通过版本门禁与迁移脚本处理必要升级。

## 6. 阶段门禁（Go / No-Go）

每个 Epoch 进入下一阶段前必须同时满足：

1. 当前 Epoch 退出条件全部通过。
2. 安全与隐私红线（本地存储、Kill Switch、高危确认）无回归。
3. 性能基线无明显退化（空闲 CPU 占用可控）。
4. 需求文档、架构文档、开发文档三者追踪矩阵无断链。
